# Graph/Flow-Based Feasible-Mission Formulation

## 1. Modeling Perspective

This formulation uses a different abstraction from the classical node-arc
MILP. Instead of deciding whether a drone travels on each individual arc, the
model first constructs a graph of complete feasible drone missions. A mission
is a full route that starts at the depot, visits an ordered subset of
customers, respects payload capacity, respects battery capacity, avoids
no-fly zones, and returns to the depot.

The optimization model then selects a collection of complete missions so that
every customer is covered exactly once. Therefore, the decision variable does
not mean "use arc \((i,j)\)"; it means "select this entire feasible drone
mission." This is a higher-level graph/flow perspective and is not a direct
one-to-one reformulation of the node-arc MILP.

This abstraction is especially useful for metaheuristics, column-generation
ideas, and route-pool based exact methods, because route feasibility is handled
before the master selection problem.

## 2. Feasible-Mission Graph

For each drone \(k \in K\), define a mission graph:

\[
G_k = (V_k, E_k).
\]

Each node \(r \in V_k\) represents one complete feasible mission for drone
\(k\). A mission has the form:

\[
r = (0, i_1, i_2, ..., i_p, 0),
\]

where \(i_1, i_2, ..., i_p \in C\) are distinct customers.

A mission belongs to \(V_k\) only if it satisfies all route-level restrictions:

- the route starts at the depot,
- the route returns to the depot,
- every customer inside the mission is visited once within that mission,
- the sum of package demands in the mission does not exceed \(Q_k\),
- total mission energy does not exceed \(B_k\),
- each travel segment avoids all no-fly zones,
- optional collision constraints can be checked during mission generation or
  added later through compatibility edges.

Edges in \(E_k\) are not travel arcs. They can represent compatibility between
missions, precedence relations, or transitions in a route-pool construction
procedure. For the basic set-partitioning version, only the mission set
\(V_k\) is required.

## 3. Sets

Let:

- \(C = \{1, 2, ..., n\}\): set of customers.
- \(K = \{1, 2, ..., m\}\): set of drones.
- \(R_k\): set of feasible complete missions available to drone \(k\).
- \(R = \bigcup_{k \in K} R_k\): set of all feasible missions.
- \(C(r) \subseteq C\): set of customers served by mission \(r\).
- \(K(r)\): drone associated with mission \(r\).

The set \(R_k\) is generated by solving or enumerating a resource-constrained
route-generation problem for drone \(k\).

## 4. Parameters

For each customer \(i \in C\):

- \(q_i\): package demand of customer \(i\).

For each drone \(k \in K\):

- \(Q_k\): payload capacity.
- \(B_k\): battery capacity.

For each mission \(r \in R\):

- \(c_r\): total energy consumption of mission \(r\).
- \(a_{ir}\): customer-coverage coefficient:

\[
a_{ir} =
\begin{cases}
1, & \text{if mission } r \text{ serves customer } i,\\
0, & \text{otherwise.}
\end{cases}
\]

- \(g_{kr}\): drone-compatibility coefficient:

\[
g_{kr} =
\begin{cases}
1, & \text{if mission } r \in R_k,\\
0, & \text{otherwise.}
\end{cases}
\]

For optional collision modeling:

- \(h_{rs}\): compatibility coefficient between missions \(r\) and \(s\).
  It equals 1 if both missions can be flown simultaneously without conflict,
  and 0 otherwise.

## 5. Mission Cost Calculation

For a mission

\[
r = (0, i_1, i_2, ..., i_p, 0),
\]

let \(d_{uv}\) be the Euclidean distance between consecutive nodes \(u\) and
\(v\). Let:

\[
L_0(r) = \sum_{\ell=1}^{p} q_{i_\ell}
\]

be the initial payload at departure. Before traveling from \(i_\ell\) to
\(i_{\ell+1}\), the remaining payload is:

\[
L_\ell(r) = \sum_{s=\ell+1}^{p} q_{i_s}.
\]

The energy cost of mission \(r\) is:

\[
c_r =
d_{0i_1}(\alpha + \beta L_0(r))
+
\sum_{\ell=1}^{p-1}
d_{i_\ell i_{\ell+1}}(\alpha + \beta L_\ell(r))
+
d_{i_p0}\alpha.
\]

The final return to the depot has no remaining package payload, so only the
base energy term is charged.

## 6. Decision Variables

Mission-selection variable:

\[
\lambda_r =
\begin{cases}
1, & \text{if feasible mission } r \text{ is selected},\\
0, & \text{otherwise.}
\end{cases}
\]

This is the central difference from the MILP formulation. The variable
\(\lambda_r\) selects an entire feasible route, not an individual movement
between two locations.

## 7. Objective Function

The objective is to minimize total energy over the selected missions:

\[
\min \sum_{r \in R} c_r \lambda_r.
\]

Because each \(c_r\) already includes distance, payload effect, battery
feasibility, and no-fly-zone feasibility, the master problem is compact and
focused on assignment decisions.

## 8. Constraints

### 8.1 Customer Coverage Constraint

Every customer must be covered by exactly one selected mission:

\[
\sum_{r \in R} a_{ir}\lambda_r = 1
\qquad \forall i \in C.
\]

This is the set-partitioning constraint. It prevents both missed deliveries
and duplicate service.

### 8.2 Drone Availability Constraint

Each drone can perform at most one mission:

\[
\sum_{r \in R_k} \lambda_r \leq 1
\qquad \forall k \in K.
\]

This reflects the single-route-per-drone assumption used in the project. If
multiple trips per drone are allowed, the right-hand side can be changed or
time-indexed mission scheduling constraints can be added.

### 8.3 Mission Feasibility Conditions

The following conditions are not written as master-problem constraints because
they are enforced during construction of \(R_k\). A mission \(r \in R_k\) is
generated only if:

\[
\sum_{i \in C(r)} q_i \leq Q_k,
\]

\[
c_r \leq B_k,
\]

and every segment in the mission avoids no-fly zones:

\[
(i_\ell, i_{\ell+1}) \in A
\qquad \text{for every consecutive pair in } r.
\]

Thus, the route-generation phase acts as a feasibility filter. The master
model never sees infeasible missions.

### 8.4 Optional Collision-Avoidance Compatibility

If two missions cannot be executed together because their trajectories conflict
in time or space, the following incompatibility constraint can be added:

\[
\lambda_r + \lambda_s \leq 1
\qquad \forall r,s \in R : h_{rs}=0.
\]

This form is convenient because collision avoidance is evaluated between
complete missions rather than between every pair of low-level travel arcs.

### 8.5 Binary Restrictions

\[
\lambda_r \in \{0,1\}
\qquad \forall r \in R.
\]

## 9. Route-Generation Logic

The mission set \(R_k\) can be generated using a resource-constrained search
procedure. A partial route stores:

- current node,
- visited customers,
- remaining payload capacity,
- accumulated energy,
- remaining battery capacity,
- ordered customer sequence.

Starting from the depot, the search extends a partial mission by adding an
unvisited customer if:

- the added package demand does not exceed remaining payload capacity,
- the added travel segment does not cross a no-fly zone,
- the route can still return to the depot within battery capacity.

Whenever a partial route can return to the depot, it produces a feasible
mission candidate. Dominance rules can be used to remove inferior partial
routes. For example, if two partial routes end at the same customer and have
served the same customer subset, the one with lower energy and higher remaining
capacity dominates the other.

## 10. Modeling Logic Summary

The graph/flow-based formulation separates the problem into two conceptual
levels. The first level constructs a graph or pool of complete feasible drone
missions. At this level, payload capacity, battery capacity, energy
consumption, route continuity, and no-fly-zone avoidance are checked directly
for entire routes. The second level selects a subset of those missions so that
each customer is served exactly once and each drone is used at most once.

This is different from the MILP formulation because the optimization variable
is a complete mission rather than a travel arc. The MILP builds routes through
local arc decisions and subtour-elimination constraints. The feasible-mission
model treats routes as atomic alternatives and solves a higher-level selection
problem. This makes it more natural for route-pool methods, decomposition, and
metaheuristic designs, while the MILP remains more direct for small exact
optimization instances.
